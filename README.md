 Проєкт тримає облік товарів на складі, оформлює поставки та продажі, веде резервування з пріоритетом, пропонує перезамовлення і фіксує списання браку. Усі дані живуть у памʼяті за шаровою архітектурою.

## Опис проблеми

Склад поступово обростає логічними нюансами, які важко тримати в голові. Основні з них такі.

Облік залишків. Кількість на складі ділиться на дві частини: вільний залишок і резерв під клієнтів. Якщо плутати ці числа, можна продати товар, який уже комусь обіцяний. Тому всі операції зміни залишку проходять через одну точку, яка стежить, щоб резерв ніколи не став більшим за загальний запас.

Резервування з пріоритетом. На один і той самий товар може претендувати кілька клієнтів. Звичайний клієнт стоїть у черзі позаду срібного, срібний позаду золотого, золотий позаду VIP. Якщо двоє з однаковим статусом, діє FIFO: перший подав запит, перший і отримав. Алгоритм розподілу пробігає по чергах і виділяє товар тим, хто може його отримати повністю; великі замовлення можуть пропуститись наперед менших, якщо вільного товару не вистачає.

Знижки. Сума продажу проходить через стратегію знижки. Базова стратегія дає знижку за рівнем лояльності, додаткова рахує знижку за обсяг чека. Стратегію можна замінити без правок у сервісі продажу.

Перезамовлення. Коли вільний залишок падає нижче мінімуму, система вмикає сповіщення і дозволяє побудувати план закупівлі. Обсяг рахується стратегією: фіксована партія, мінімум-максимум або класична формула EOQ.

Списання браку. Зіпсований чи втрачений товар треба зняти з обліку і одночасно зафіксувати фінансовий збиток. Різні причини списання дають різний коефіцієнт залишкової вартості.

## Актори та сценарії

У системі діють дві ролі.

**Менеджер складу** заводить товари, працює з постачальниками, проводить поставки, списує брак, будує план перезамовлення.

**Клієнт або продавець** оформлює резерв і продаж, переглядає залишки.

Ключові сценарії:

1. Реєстрація товару і прийом поставки. Менеджер створює постачальника, заводить новий товар, оформлює поставку, додає позиції, підтверджує і приймає її на склад. Залишок зростає.
2. Резервування товару клієнтом. Клієнт надсилає заявку, система ставить її в чергу за статусом, далі при найближчій вільній кількості резерв виділяється.
3. Продаж зі знижкою. Продавець створює продаж, додає позиції, обирає рівень клієнта. Стратегія знижки рахує фінальну суму, товар списується із залишку.
4. Видача зарезервованого товару. Клієнт забирає замовлення, резерв переходить у фінальний продаж, а сповіщення про наявність відсилається наступному в черзі.
5. Списання браку. Менеджер фіксує брак, кількість падає, сума збитку зростає з урахуванням залишкової вартості.
6. План перезамовлення. Менеджер запитує план, отримує перелік товарів, які впали нижче мінімуму, з рекомендованою партією за обраною стратегією.
7. Зняття товару з продажу. Менеджер позначає товар як discontinued, якщо на ньому немає активного резерву.

Діаграми у форматі Mermaid лежать у `docs/diagrams/`: use case, доменна модель і діаграма класів.

## Архітектура

Код розкладений по шарах і не знає про БД чи мережу.

```
src/
  models/       сутності предметної області
  storage/      інтерфейс IRepository і реалізація на словнику
  services/     прикладна логіка
  patterns/     шаблони проєктування
  utils/        винятки, переліки, валідатори, генератор id
```

`InventoryService` єдиний має право змінювати залишок. Інші сервіси проходять через нього, тому правила резерву і мінімального залишку перевіряються в одному місці. Сторонні залежності передаються через конструктор. Сама збірка зібрана у класі `Application`, який дає готовий контейнер для головного сценарію і тестів.

In-memory сховище схований за інтерфейсом `IRepository`. Сервіси не знають, як саме воно влаштоване, тому замінити словник на реальну базу можна без правок бізнес-логіки.

## Шаблони проєктування

**Strategy.** `DiscountStrategy` має реалізації `NoDiscount`, `TierDiscount`, `BulkDiscount`. `ReorderStrategy` має `FixedQuantityStrategy`, `MinMaxStrategy`, `EOQStrategy`. Зміна стратегії не вимагає правок у сервісах.

**Observer.** `StockSubject` тримає список спостерігачів, `StockObserver` визначає інтерфейс події. Готові реалізації: `LowStockNotifier`, `ReorderLogger`, `ReservationAvailabilityNotifier`. `InventoryService` сам пушить події, коли залишок опускається до мінімуму чи виділяється резерв.

**Factory.** `MovementFactory` створює записи руху товару під різні операції (прихід, відвантаження, резерв, зняття резерву, списання, коригування). Сервіс не знає, як саме зібрати запис, лише просить фабрику.

## SOLID

- Single Responsibility. Кожен клас закриває одну роль: модель тримає інваріант, сервіс координує, сховище зберігає.
- Open/Closed. Нову знижку чи стратегію перезамовлення можна додати окремим класом, не торкаючись існуючих.
- Liskov. Усі стратегії та спостерігачі взаємозамінні через свій абстрактний клас.
- Interface Segregation. `IRepository` тримає лише потрібний набір операцій, конкретні репозиторії додають свої вибірки.
- Dependency Inversion. Сервіси приймають залежності через конструктор, не створюють їх самі. Бачимо у `Application.__init__`.

## ШІ-контекст

- `.cursorrules` — глобальні правила: ніяких зовнішніх БД чи API, тести разом з кодом, реалізація сховища лише після інтерфейсу, винятки тільки з `src/utils/exceptions.py`.
- `.cursor/rules/architecture.md` — опис шарів та потоку даних, щоб агент не плутав, де живе яка логіка.
- `.cursor/rules/testing.md` — інструкція щодо генерації тестів, використання моків та конфігурації звітів покриття.

## Запуск

Потрібен Python 3.13, на 3.10–3.12 теж працює.

```
pip install -r requirements.txt
python main.py
```

Файл `main.py` показує наскрізний сценарій: реєстрація постачальника і товарів, поставка, резерв за пріоритетом, продаж зі знижкою, списання, побудова плану перезамовлення.

## Тести та покриття

```
python -m pytest
```

Після запуску в теці `reports` зʼявляється:

- `coverage.xml` у форматі Cobertura для SonarQube.
- `junit.xml` для звітів про тести.
- `htmlcov/index.html` для перегляду непокритих рядків у браузері.

Зараз у проєкті 284 тести, покриття 98 відсотків. Кожна гілка валідації, кожен перехід між станами документа та кожна стратегія мають свій тест. Сервіси покриті як інтеграційно через `Application`, так і ізольовано через `unittest.mock`.

## CI/CD

Пайплайн у `.github/workflows/ci-pipeline.yml` на кожен пуш і pull request виконує:

1. Збірку та встановлення залежностей.
2. Запуск тестів з генерацією звітів.
3. Завантаження артефакта `test-reports`.
4. Сканування SonarCloud за `sonar-project.properties`.
